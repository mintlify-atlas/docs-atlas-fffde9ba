---
title: "Multi-vehicle coordination"
description: "Coordinate multiple vehicles using ZMQ: start the proxy, use ZmqStateMachine, expose states and fields, and trigger remote transitions."
---

aerpawlib's ZMQ layer lets independent vehicle scripts communicate without direct network connections between scripts. A central proxy relays messages; each script only needs to know the proxy's address.

## Architecture

```
  Leader script          ZMQ Proxy            Follower script
  (vehicle 1)        aerpawlib --run-proxy     (vehicle 2)
       │                      │                      │
       │──── TRANSITION ────▶ │ ──── TRANSITION ────▶│
       │◀─── FIELD_CALLBACK ──│ ◀─── FIELD_REQUEST ──│
```

- Each script connects to the proxy using a unique `--zmq-identifier`.
- The proxy routes `TRANSITION` and `FIELD_REQUEST`/`FIELD_CALLBACK` messages by identifier.
- Scripts do not need direct connectivity to each other.

---

## Setup

<Steps>
  <Step title="Start the ZMQ proxy">
    Run the proxy in its own terminal before starting any vehicle scripts:

    ```bash
    aerpawlib --run-proxy
    ```

    The proxy listens on default ZMQ ports and routes messages between all connected runners.
  </Step>

  <Step title="Start the leader script">
    ```bash
    aerpawlib \
      --script examples/v1/zmq_runner/leader.py \
      --conn udp://127.0.0.1:14550 \
      --vehicle drone \
      --zmq-identifier leader \
      --zmq-proxy-server 127.0.0.1
    ```
  </Step>

  <Step title="Start the follower script">
    ```bash
    aerpawlib \
      --script examples/v1/zmq_runner/follower.py \
      --conn udp://127.0.0.1:14551 \
      --vehicle drone \
      --zmq-identifier follower \
      --zmq-proxy-server 127.0.0.1
    ```

    Each vehicle needs a unique `--zmq-identifier`. The scripts start independently and synchronize through state transitions.
  </Step>
</Steps>

<Warning>
  Always start the ZMQ proxy before the vehicle scripts. Scripts that cannot reach the proxy at startup will fail.
</Warning>

## Leader/follower pattern

The canonical pattern: the leader controls mission sequencing and tells the follower when to transition between states.

<Tabs>
  <Tab title="v1">
    **Leader** (`examples/v1/zmq_runner/leader.py`):

    ```python
    from aerpawlib.v1.runner import ZmqStateMachine, state, sleep

    class LeaderRunner(ZmqStateMachine):
        @state(name="launch", first=True)
        async def state_start(self, _):
            print("waiting to start")
            await sleep(10)
            return "start_ping"

        @state(name="start_ping")
        async def state_start_ping(self, _):
            # ... measure ping latency ...
            await self.transition_runner("follower", "takeoff")
            return "wait_for_waypoint"

        @state(name="wait_for_waypoint")
        async def state_wait_waypoint(self, _):
            return "wait_for_waypoint"  # spin until follower triggers next state

        @state(name="waypoint_ping")
        async def state_ping_waypoint(self, _):
            await self.transition_runner("follower", "rtl")
            return "wait_for_rtl"

        @state(name="wait_for_rtl")
        async def state_wait_rtl(self, _):
            return "wait_for_rtl"

        @state(name="last_ping")
        async def state_last_ping(self, _):
            await self.transition_runner("follower", "land")
            return  # end
    ```

    **Follower** (`examples/v1/zmq_runner/follower.py`):

    ```python
    from aerpawlib.v1.runner import ZmqStateMachine, state, sleep
    from aerpawlib.v1.util import Coordinate, VectorNED
    from aerpawlib.v1.vehicle import Drone

    class FollowRunner(ZmqStateMachine):
        @state(name="launch_wait", first=True)
        async def state_start(self, _):
            return "launch_wait"  # spin until leader sends "takeoff"

        @state(name="takeoff")
        async def state_takeoff(self, drone: Drone):
            print("taking off")
            await drone.takeoff(10)
            return "fly_to_waypoint"

        @state(name="fly_to_waypoint")
        async def state_waypoint(self, drone: Drone):
            await drone.goto_coordinates(drone.position + VectorNED(20, 0))
            await sleep(5)
            await self.transition_runner("leader", "waypoint_ping")
            return "waypoint_wait"

        @state(name="waypoint_wait")
        async def state_wait_waypoint(self, _):
            return "waypoint_wait"

        @state(name="rtl")
        async def state_rtl(self, drone: Drone):
            home_coords = Coordinate(
                drone.home_coords.lat, drone.home_coords.lon, drone.position.alt
            )
            await drone.goto_coordinates(home_coords)
            await sleep(5)
            await self.transition_runner("leader", "last_ping")
            return "wait_last_ping"

        @state(name="land")
        async def state_land(self, drone: Drone):
            await drone.land()
            return  # end
    ```

## ZMQ decorators

### `@expose_zmq(name)`

Mark a `@state` or `@timed_state` method so that another runner can trigger a transition into it remotely.

```python
@state(name="takeoff")
@expose_zmq("takeoff")
async def state_takeoff(self, drone: Drone):
    await drone.takeoff(10)
    return "hover"
```

<Warning>
  `@expose_zmq` must be applied to a method that also has `@state` or `@timed_state`. Applying it to a plain method raises an error.
</Warning>

### `@expose_field_zmq(name)`

Make a method's return value queryable by another runner via `query_field`.

```python
from aerpawlib.v1.runner import expose_field_zmq

@expose_field_zmq("position")
async def get_position(self, vehicle):
    return vehicle.position
```

### `transition_runner(identifier, state_name)`

Send a state-transition command to another runner. The target runner must have exposed that state with `@expose_zmq`.

```python
await self.transition_runner("follower", "takeoff")
```

### `query_field(identifier, field, timeout)`

Request a field value from another runner and wait for the reply.

```python
position = await self.query_field("follower", "position", timeout=5.0)
```

## Tips

- Each vehicle process must have a **unique** `--zmq-identifier`. Duplicate identifiers cause message routing conflicts.
- Spin states (returning to the same state name) are the standard way to wait for a remote transition.
- The proxy is stateless; it does not buffer messages. Start all scripts promptly after the proxy so no messages are dropped during startup.
- For more than two vehicles, add additional runner scripts with unique identifiers — the proxy handles any number of participants.
